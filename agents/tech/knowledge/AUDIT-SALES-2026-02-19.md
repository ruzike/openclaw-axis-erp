## ✅ ОБОСНОВАНИЕ РАЗДЕЛЕНИЯ

### Почему разделение НЕ навредит

**1. Прямая коммуникация**
```
Маркетинг → Клиент → Коммерческий → Finance
```
Быстрее и прозрачнее: Клиент сразу получает КП, а не промежуточные сообщения.

**2. Глубокая специализация**
- Маркетинг-агент: лучший в лидогенерации (создаёт контент, следит за каналами)
- Коммерческий агент: лучший в закрытии сделок (знает математику, продукт, конкурентов)

**3. Меньше когнитивной нагрузки**
- Маркетинг-агент: думает только о клиентах
- Коммерческий агент: думает только о деньгах и сделках

**4. Легче масштабировать**
- Если маркетинг вырастет → создадим 2 маркетинга (интернет + соцсети)
- Если коммерция вырастет → создадим 2 коммерка (розница + B2B)

**5. Меньше конфликтов**
- Клиенты не будут говорить: "Долго отвечаешь на сообщения" (коммерческий агент)
- Коммерческий агент не будет говорить: "Ты не приводишь достаточного количества качественных лидов" (маркетинг-агент)

### Почему разделение ВАЖНО

**1. Контекстное переключение**
- В одном SOUL.md (286 строк) смешаны: "Напиши клиенту: Привет! Как дела?" и "Рассчитай стоимость: 10 ракурсов × $1,500 + наценка 20%"
- Это создает когнитивный шум

**2. Риск галлюцинаций**
- Агент начинает предлагать клиентам скидки (Level 3 decision) и одновременно обещать сжатые сроки (маркетинг)
- Разные агенты с разными моделями и температурами будут лучше придерживаться своей зоны ответственности

**3. Упрощение onboarding**
- Нового сотрудника проще обучить: "Вот твой паттерн общения с клиентами" vs "Вот твой стиль общения с клиентами и вот твой стиль общения с агентами Finance"
- Опытному сотруднику проще: "Я — продавец-максимализм" vs "Я — комплексный менеджер продаж"

### Шаг 3: Маршрутизация (1 день)

#### 3.1. Настроить интеграцию Trello
```python
# Trello Butler Rule: New Lead → Auto-assign to Marketing Agent
{
  "action": "updateItemSpecifics",
  "rule": {
    "type": "cardDueDate",
    "condition": {
      "field": "due",
      "operator": "is"
    }
  },
  "value": {
    "labels": ["marketing"]
  }
}
```

#### 3.2. Настроить автоматическую передачу
- **Marketing → Клиент → Deals:** Когда клиент выразил интерес, Deals получает тег "квалифицирован"
- **Deals → Finance:** Когда сделка закрыта, Finance получает тег "оплачено"
- **Deals → Ops:** Когда клиент передаётся в производство, Ops получает тег "в работе"

## 🚨 РИСКИ И МИННИМУМЫ

### Риски

1. **Разрыв коммуникации**
   - Риск: Клиент не понимает, куда обращаться
   - Минимум: Создать единый контакт-бот с маршрутизацией

2. **Потеря контекста**
   - Риск: New agent не знает контекста сделки
   - Минимум: Создать shared state (Trello) с полными данными

3. **Проблемы с моделями**
   - Риск: Marketing-агент может ошибаться в расчётах
   - Минимум: Проверять все расчёты перед отправкой клиенту

### Минимумы (что НЕ удалять)

- Shared knowledge (источники лидов, продукты, компании)
- Система мотивации (KPI, бонусы)
- Интеграции с существующими инструментами
